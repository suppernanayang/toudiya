# 项目约束卡

每次改动前，先用中文说明：打算改哪几处、怎么改、风险是什么、怎么验收。用户确认“可以”之前，不要修改业务代码。

## 产品硬规则

- 产品定位：带人工审核台的 AI 求职助手，覆盖简历定制、岗位分析、辅助投递、官网填表、投递追踪和面试准备。
- 核心闭环：导入岗位/JD -> 分析 JD -> 匹配或定制简历 -> 审核台确认 -> 辅助投递/填表 -> 追踪和面试准备。
- 任何真实投递、官网提交、账号登录、平台操作、自动投递动作，都必须经过用户人工确认。
- AI 可以重组、排序、改写和强化用户真实经历，但不允许编造实习、项目、技能、公司、学校、证书或成果数据。
- 不确定的数据必须标为“待用户确认”，不能直接写入最终简历。
- 简历版本默认全部留痕保存，不覆盖旧版；用户可删除、下载、重命名、复制为新版本或设为岗位类型默认简历。

## 审核台规则

- 审核台必须支持：直接采用 AI 定制版、平台内编辑、下载后外部修改并重新上传最终版。
- 每个岗位必须绑定最终投递版本，记录当时 JD、简历文件、投递话术、投递时间和投递状态。
- 审核台需允许用户选择：是否投递、使用哪份简历、是否为该岗位单独定制、是否上传外部最终版、投递优先级和备注。
- 用户上传的最终版优先级高于平台内编辑版，平台内编辑版优先级高于 AI 草稿版，除非用户手动切换。

## 技术和范围限制

- 只能改动用户确认过的目录或文件。
- 改数据库结构、删文件、扩大改动范围、引入新依赖、执行 git commit 前必须先问并得到同意。
- 不先做绕过验证码、反爬、批量海投、未经确认的自动提交。
- 涉及账号、隐私、简历、联系方式、投递记录、第三方平台登录时，必须先说明数据会如何读取、存储和使用。

## 工作方式

- 非平凡改动开始前，先输出：【你的理解】【方案】【风险】【验收】，等待用户确认。
- 修 bug 必须先写能复现问题的测试或最小复现场景，再改业务代码。
- 不确定怎么做时，先给 2 个方案和各自风险，让用户选择。
- 出错时，原样给出关键报错、返回内容和证据，不要脑补。
- 多次修不好时，停止改业务逻辑，进入诊断模式，只收集证据。

## 完成标准

- 改动只落在允许范围内。
- 相关测试或冒烟验证通过。
- 相关功能手动验证没坏。
- 最后说明改了哪些文件、为什么改、如何验收。
