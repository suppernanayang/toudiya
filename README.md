# 投递鸭 Toudiya

带人工审核台的 AI 求职助手：JD 分析、简历定制、辅助投递、投递追踪、面试准备一条龙，AI 负责脏活累活，人负责最后拍板。

> 📄 完整产品介绍（痛点、使用流程、技术实现细节）：[《投递鸭 —— 带人工审核台的 AI 求职助手》](https://docs.xiaohongshu.com/doc/63fff8070a8c46c23fe232b1395e5bc0)

## 这是什么

市面上大部分"AI 求职工具"追求的是"帮你自动海投几百家"，投递鸭刻意不做这件事。它的核心逻辑是反过来的：**AI 负责拆解 JD、整理经历、生成初稿、准备话术，人负责所有会影响真实结果的判断和动作**——简历改得对不对、要不要投、要不要真的点提交，这些都必须由用户自己确认。

单用户本地运行，不依赖任何云端账号体系，数据都在自己的电脑上。

## 核心流程

```
① 导入岗位JD  →  ② AI分析JD  →  ③ AI定制简历  →  ④ 人工审核确认
                                                          ↓
⑦ 面试准备  ←────  ⑥ 投递追踪  ←──────────────────  ⑤ 辅助投递
```

- **导入 JD**：手动粘贴，或用浏览器插件在招聘详情页一键识别抓取
- **AI 分析 JD**：拆解成职责、要求、关键词、匹配风险点
- **AI 定制简历**：基于经历库 + JD 分析结果生成定制稿，只重组/改写已有经历，不编造
- **人工审核确认**：审核台展示改动摘要、风险提示、待确认清单，用户拍板后才能进入下一步
- **辅助投递**：确认最终版简历和话术后，用户自己去目标平台完成真正的投递动作
- **投递追踪**：记录每次投递的简历版本、时间、状态，持续跟进进度
- **面试准备**：基于最终投递材料生成自我介绍、STAR 答案框架、可能追问、复习清单

## 功能模块

| 模块 | 路由 | 说明 |
|---|---|---|
| 工作台 | `/dashboard` | 待办总览、快捷入口、模型配置状态 |
| 简历库 | `/resumes` | 简历导入、格式化、多版本管理（全部留痕，不覆盖旧版） |
| 经历库 | `/experiences` | 简历定制唯一的素材来源，AI 只能引用这里已有的内容 |
| 岗位池 | `/jobs` | 岗位导入与状态流转管理 |
| 审核台 | `/review` | 核心确认节点：简历版本对比、风险提示、投递决策 |
| 投递追踪 | `/applications` | 投递记录与进度更新 |
| 面试准备 | `/interviews` | 基于最终投递材料生成面试准备包 |
| 浏览器插件 | `/extension` | 插件配对状态管理，见下方"浏览器插件"一节 |

## 怎么解决"AI 瞎编"这个问题

所有 AI 任务共用同一套系统级别的硬性规则（`src/lib/llm/prompts.ts`）：

- 只能基于用户提供的简历原文、经历库内容、JD 原文来生成内容，不能编造公司、学校、项目、技能、证书或成果数据
- 任何用户没有明确提供、但看起来"应该有"的数据（具体百分比、金额、排名），不允许直接写进正式结果，必须放进待确认清单
- 每次 AI 输出必须同时返回结果、风险提示（`risk_notes`）、待确认清单（`pending_confirmations`）、出处标注（`source_references`），这三个字段是所有任务返回结构里的强制字段，没有内容也要显式返回空数组

## 技术栈

- **框架**：Next.js（App Router）+ TypeScript + Tailwind CSS
- **数据层**：Prisma + SQLite
- **AI**：LLM Provider 抽象层，支持 DeepSeek / OpenAI 切换，任务级别可单独配置模型和自动降级
- **PDF 简历导出**：`@react-pdf/renderer` + 思源黑体（Noto Sans SC），支持自动缩放到 1 页、导出前实时预览调整字号
- **浏览器插件**：`extension/` 独立子项目，Manifest V3 + Vite + React，插件与本地服务之间自动配对（无需手动复制密钥），JD 识别采用"平台精确规则 + AI 通用兜底"两层策略

## 本地运行

```bash
# 1. 安装依赖
npm install

# 2. 配置环境变量
cp .env.example .env
# 编辑 .env，至少填一个 DEEPSEEK_API_KEY 或 OPENAI_API_KEY

# 3. 初始化数据库
npm run db:migrate

# 4. 启动开发服务器
npm run dev
```

访问 `http://localhost:3000` 即可使用。

其他常用脚本：

```bash
npm run lint      # eslint 检查
npm run smoke     # 冒烟测试（检查数据库连接、storage 目录、关键页面可访问性）
npm run db:studio # 打开 Prisma Studio 查看数据库
```

## 浏览器插件

插件源码在 `extension/` 目录，独立的 Vite + React 子项目，不影响主项目构建。构建产物已经提交进仓库（`extension/dist`），克隆下来直接在应用内 `/extension` 页面点击"下载插件安装包"就能拿到打包好的 zip，不需要自己装依赖跑构建。解压后在 Chrome 的 `chrome://extensions` 开发者模式下"加载已解压的扩展程序"即可安装，安装后打开 `/extension` 页面会自动完成插件与本地服务的配对。

如果要修改插件代码，改完需要重新构建：

```bash
cd extension
npm install
npm run build
```

## 项目原则

- **真实投递、账号登录、平台操作、自动提交必须经过人工确认**——不做、也不会做"自动帮用户完成投递"这件事
- **AI 只能重组、排序、改写用户真实经历，不能编造**没有的实习、项目、技能、公司、学校、证书或成果数据
- **不确定的数据必须标为"待确认"**，不能直接写入最终简历
- **简历版本全部留痕保存，不覆盖旧版**，用户可以随时删除、下载、重命名、复制成新版本
